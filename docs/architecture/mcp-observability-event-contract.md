# Omvra MCP observability event contract

Status: implementation contract for `omvra.mcp.audit.v1`

This contract defines the metadata persisted for MCP actions. It is an allow-list: fields not listed here must not be persisted in an audit event. Request and response payloads remain outside the audit boundary.

## Event shape

```json
{
  "schemaVersion": 1,
  "auditId": "audit-...",
  "timestamp": "2026-07-15T10:00:00.000Z",
  "startedAt": "2026-07-15T10:00:00.001Z",
  "finishedAt": "2026-07-15T10:00:00.024Z",
  "durationMs": 23,
  "type": "mcp_tool_call",
  "agent": "codex",
  "clientName": "Codex",
  "clientVersion": null,
  "toolName": "tasks.update",
  "transport": "http",
  "origin": "127.0.0.1",
  "target": {
    "taskId": "task-...",
    "projectId": "1",
    "entityId": null
  },
  "outcome": "success",
  "failureClass": null
}
```

## Field rules

| Field | Type | Source and missing value | Privacy rule |
| --- | --- | --- | --- |
| `schemaVersion` | `1` | Contract constant | Required; enables future normalizers |
| `auditId` | string | Generated by the existing audit writer | Safe identifier only |
| `timestamp` | ISO string | Existing event creation time | Safe metadata |
| `startedAt`, `finishedAt` | ISO string | Shared dispatch boundary | Safe metadata; both required after dispatch begins |
| `durationMs` | non-negative integer | `finishedAt - startedAt` | Safe timing metadata; never infer from payload |
| `type` | string | `mcp_tool_call` or existing audit type during migration | Bounded known values |
| `agent` | enum | Normalized safe client hint; `unknown` when absent/unrecognized | Values only: `codex`, `copilot`, `claude`, `opencode`, `openai`, `unknown` |
| `clientName`, `clientVersion` | string or null | Safe client metadata when explicitly supplied | Never copy arbitrary headers |
| `toolName` | string or null | Validated MCP tool name | Bounded tool/path identifier only |
| `transport` | `http` or `stdio` | Request transport; `http` fallback for legacy events | No socket or connection dump |
| `origin` | string or null | Redacted local origin/transport label | Do not persist authorization, query strings, or raw forwarded headers |
| `target.taskId`, `target.projectId`, `target.entityId` | string or null | Extracted from validated arguments/results | Identifiers only; no titles, descriptions, comments, or payload fragments |
| `outcome` | enum | Dispatcher result | Values: `success`, `failure`, `denied` |
| `failureClass` | enum or null | Stable classification for non-success outcomes | Values: `unauthorized`, `forbidden`, `invalid_params`, `not_found`, `conflict`, `internal`, `unknown`; no stack traces or error bodies |

Existing fields such as `reason`, `capabilityProfile`, revision metadata, and status identifiers may remain during migration only when they are individually allow-listed and contain no payload content. New code must not pass arbitrary `args`, `result`, `headers`, or error objects to the audit writer.

## Provenance normalization

Provenance is derived from an explicitly supported client hint at the transport boundary. Matching is case-insensitive and should recognize the supported product names and common safe aliases. Missing, malformed, or unsupported values normalize to `unknown`. Provenance is descriptive metadata, not an authorization signal.

HTTP may use a safe client header or negotiated client metadata. Stdio may use the launcher-provided client metadata. Neither transport persists the full header set, command line, environment, access token, or connection object.

## Compatibility and migration

- Keep the storage key `omvra.mcp.audit.v1`.
- Legacy entries are read through a normalizer that supplies `schemaVersion: 1`, maps the existing `timestamp` to missing timing fields only when no better timing exists, and fills new dimensions with `null` or `unknown`.
- New writes use the normalized allow-list shape.
- Existing consumers may continue reading legacy fields while the IPC and renderer projections migrate to the normalized fields.
- No raw payload backfill is permitted.

## Explicitly excluded data

Never persist MCP request arguments, response bodies, prompt/resource contents, access tokens, authorization headers, complete user-agent strings when they contain identifying details beyond the approved client fields, stack traces, filesystem paths, task titles/descriptions/comments, or arbitrary error objects.
